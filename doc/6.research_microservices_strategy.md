# Research: Database Strategy & Inter-Service Communication
**Purpose:** Microservices migration planning for Personal Finance app
**Author:** Claude (Research Agent)
**Date:** 2026-03-22

---

## Executive Summary

For a personal finance app with financial ACID requirements and a team of 1–3 developers, the practical answer is: **Shared PostgreSQL with multiple schemas → sync REST → simple service boundary**. Over-engineering with Kafka, gRPC, and per-service databases would cost more to operate than it saves. This document justifies that opinion.

---

## 1. Database Strategy

### Comparison

| Criteria | Shared DB (Multi-Schema) | DB per Service | Hybrid (RW + Read Replica) |
|---|:---:|:---:|:---:|
| **ACID for financial data** | ✅ Strong | ⚠️ Needs distributed txn | ✅ |
| **Migration complexity** | ✅ Low — minimal change | ❌ High — duplication problem | ⚠️ Medium |
| **Data duplication** | None | User data must replicate | None for writes |
| **Fault isolation** | ❌ Shared blast radius | ✅ Per-service | ⚠️ Partial |
| **Operational overhead** | ✅ Single DB to manage | ❌ 3–5 DB instances | ⚠️ 2 instances |
| **Cross-service JOINs** | ✅ Native SQL | ❌ App-level denormalization | ✅ Direct for writes |
| **Independent scaling** | ❌ Single instance | ✅ Per service | ⚠️ Partial |

### Analysis by Option

**Option A — Shared DB (Multiple Schemas)** is the right starting point:

```
┌─────────────────────────────────────────────┐
│  PostgreSQL Instance (single container/VM)  │
│                                             │
│  ┌──────────┐ ┌───────────┐ ┌───────────┐  │
│  │ auth.    │ │ core.     │ │ analytics.│  │
│  │ users    │ │ transactions│ │ dashboard │  │
│  │ sessions │ │ categories │ │ reports   │  │
│  │ accounts │ │ wallets    │ └───────────┘  │
│  │ verify   │ │ budgets    │                │
│  └──────────┘ │ assets     │                │
│               └───────────┘                │
└─────────────────────────────────────────────┘
```

Each domain gets its own schema (`auth`, `core`, `analytics`). Foreign keys cross schemas via fully-qualified names (`auth.users.id`). Single PostgreSQL instance is battle-tested for this pattern at small-to-medium scale.

**Option B — Database per Service** adds real problems:
- `auth` service owns `users` → `core.transactions` has `user_id` → user must exist in `auth` DB. Sync mechanism needed (or a shared user service everyone calls).
- Distributed transactions (e.g., "create user + create profile + seed categories") become saga/2PC nightmares.
- 3–5 PostgreSQL containers to manage, monitor, back up, and upgrade.
- For a team of 1–3, this is a significant operational burden with zero immediate benefit.

**Option C — Hybrid (Primary + Read Replica)** is a future optimization, not a migration target. Keep this in mind for Phase 3 when reporting/analytics queries start impacting transaction writes.

### Recommendation

> **Use Shared PostgreSQL with per-domain schemas (Option A).** This is the standard "strangler fig" migration path from SQLite. Start with one big schema (`app`) if multi-schema feels premature, but design FK references to allow schema splitting later without data movement.

## 2. SQLite → PostgreSQL Migration

### What Changes in Application Code

Drizzle ORM abstracts most of the differences. Key changes:

| Area | SQLite | PostgreSQL | Code Impact |
|---|:---|:---|:---|
| **UUID generation** | `crypto.randomUUID()` | Same (works at app layer) | ✅ No change |
| **Timestamps** | `sql`\`datetime('now')\`` | `sql\`now()\`` or `defaultNow()` | Drizzle handles via `$defaultFn` |
| **Integers (cents)** | `integer` | `integer` | ✅ No change |
| **Text** | `text` | `text` | ✅ No change |
| **Unique constraints** | `uniqueIndex` | `uniqueIndex` (or `unique()`) | ✅ Drizzle syntax same |
| **Auto-increment IDs** | `text` + UUID | Same | ✅ UUID everywhere |
| **Foreign keys** | `references()` | `references()` | ✅ Drizzle syntax same |
| **Boolean** | `integer` (0/1) | `boolean` | `mode: 'boolean'` in schema → Drizzle handles |
| **Array/JSON** | `text` (JSON string) | `jsonb` or `text` | Keep as `text`/`json` for simplicity |
| **ILIKE/LIKE** | `LIKE` (case-insensitive for ASCII) | `ILIKE` for case-insensitive | Search uses JS `.filter()` anyway |
| **Date arithmetic** | `date LIKE 'YYYY-MM-%'` | `date >= 'YYYY-MM-01' AND date < 'YYYY-MM-01'::date + interval '1 month'` | Update date range queries |
| **Transaction wrappers** | `db.transaction()` | `db.transaction()` | Same Drizzle API |

**Minimal changes expected:**
- `better-auth` config may need adapter swap (SQLite → PostgreSQL)
- Batch of raw SQL queries (date filters, search) need PostgreSQL syntax updates
- No ORM-level changes — Drizzle `drizzle-orm/sqlite-core` → `drizzle-orm/postgres-core`

### Migration Strategy: Expand-Contract

```
Phase 1: Dual-write (expand)
  ┌──────────────────────────────────┐
  │ Write to both SQLite + Postgres │
  │ Read from SQLite only           │
  │ Background sync copies all data  │
  └──────────────────────────────────┘
          ↓
Phase 2: Read replica validation
  ┌──────────────────────────────────┐
  │ Compare SQLite ↔ Postgres data   │
  │ Fix discrepancies               │
  │ Run smoke tests on Postgres reads │
  └──────────────────────────────────┘
          ↓
Phase 3: Switch reads (contract)
  ┌──────────────────────────────────┐
  │ Read from Postgres              │
  │ Write to Postgres only          │
  │ Delete SQLite after 30 days     │
  └──────────────────────────────────┘
```

**Without downtime** is achievable because this is a single-user personal app — the "blue-green" constraint is less critical than for a SaaS product. The expand-contract pattern above is sufficient.

## 3. Inter-Service Communication

### Comparison

| Pattern | Coupling | Consistency | Complexity | Latency |
|---|:---:|:---:|:---:|:---:|
| **Sync REST HTTP** | Medium | Strong (ACID) | Low | Low |
| **gRPC + Protobuf** | Low | Strong | High | Very Low |
| **Message Queue (Kafka/RabbitMQ)** | Very Low | Eventual | Very High | Variable |

### Recommendation: Sync REST with Internal HTTP Calls

```
┌──────────────┐
│  API Gateway │  ← single entry: /api/auth/*, /api/core/*, /api/analytics/*
│  (Nginx/CDN) │
└──────┬───────┘
       │ internal HTTP
 ┌─────┴─────┐  ┌──────────┐  ┌────────────┐
 │ Auth Svc  │  │ Core Svc │  │ Analytics  │
 │ (Hono)    │  │ (Hono)   │  │ (Hono)     │
 └─────┬─────┘  └─────┬────┘  └─────┬──────┘
       │              │             │
       └──────────────┴─────────────┘
                     │
              ┌──────┴───────┐
              │  PostgreSQL   │
              │  (shared DB) │
              └──────────────┘
```

**Why REST over alternatives:**

1. **Real-time requirements are low.** Dashboard updates after a transaction POST can tolerate 100–500ms. No need for WebSocket or SSE.
2. **gRPC is overkill.** Protocol buffers, `.proto` files, code generation — all overhead for internal calls that Hono's REST handlers already handle well. The perf difference is irrelevant at personal app scale.
3. **Message queues (Kafka/RabbitMQ) are premature.** They solve distributed systems problems at scale (event sourcing, fan-out to many consumers, replay). For a 1–3 person team managing 5 service types, the operational complexity (broker setup, consumer groups, dead-letter queues, monitoring) is a full-time job on its own.

**When async messaging becomes relevant:** If Analytics service grows to include push notifications, email alerts, or webhook integrations → add Redis Streams (lightweight, already deployable via Docker Compose) at that point, not upfront.

## 4. Service Discovery

### Recommendation: API Gateway with Nginx + Environment-Based URLs

For Docker Compose:

```yaml
# docker-compose.yml (target architecture)
services:
  api-gateway:         # Nginx reverse proxy
    image: nginx:alpine
    ports:
      - "3000:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro

  auth-svc:
    environment:
      - DATABASE_URL=postgres://postgres:5432/personal_finance
      - INTERNAL_CORE_URL=http://core-svc:3001   # Service-to-service
      - INTERNAL_ANALYTICS_URL=http://analytics-svc:3002

  core-svc:
    environment:
      - DATABASE_URL=postgres://postgres:5432/personal_finance
      - INTERNAL_AUTH_URL=http://auth-svc:3000

  analytics-svc:
    environment:
      - DATABASE_URL=postgres://postgres:5432/personal_finance
      - ANALYTICS_CACHE_TTL=300
```

**Service discovery via Docker Compose DNS** — `core-svc`, `auth-svc` are resolved by container name. No Kubernetes DNS, no Consul, no service mesh.

**Nginx routing rules:**
```
/api/auth/*      → auth-svc:3000
/api/core/*      → core-svc:3001
/api/analytics/* → analytics-svc:3002
```

**Single entry point advantage:** The frontend SPA only needs to know `http://localhost:3000/api`. No CORS config scattered across services. Auth token validation happens at gateway layer.

## 5. Error Handling & Resilience

### Patterns to Implement (in priority order)

**1. Timeouts (do first, always)**

Every outbound HTTP call must have a timeout. No infinite waits.

```typescript
// Pattern: abort controller + timeout
const controller = new AbortController()
const timeout = setTimeout(() => controller.abort(), 3000)

const result = await fetch(internalUrl, {
  signal: controller.signal,
  credentials: 'include',
})

clearTimeout(timeout)
```

**2. Retry with Exponential Backoff**

Only for idempotent operations (GET requests, read-side calls).

```typescript
async function fetchWithRetry(url: string, retries = 3): Promise<Response> {
  for (let attempt = 0; attempt < retries; attempt++) {
    try {
      const res = await fetch(url)
      if (res.ok) return res
      if (attempt === retries - 1) return res
    } catch (err) {
      if (attempt === retries - 1) throw err
    }
    await new Promise(r => setTimeout(r, 2 ** attempt * 100)) // 100, 200, 400ms
  }
  throw new Error('Max retries exceeded')
}
```

**3. Circuit Breaker (implement when pain emerges)**

Don't add Hystrix or a circuit breaker library upfront. Add it when you observe cascading failures in practice. A simple in-memory version:

```typescript
const circuitBreaker = {
  failures: 0,
  lastFailure: 0,
  threshold: 5,
  resetTimeout: 30_000,

  canTry(): boolean {
    if (Date.now() - this.lastFailure > this.resetTimeout) {
      this.failures = 0
    }
    return this.failures < this.threshold
  },

  recordSuccess() { this.failures = 0 },
  recordFailure() {
    this.failures++
    this.lastFailure = Date.now()
  },
}
```

**4. Bulkhead Isolation (defer)**

Only necessary when one failing service degrades another. At 3 services, this is unlikely to be a problem. Revisit when you have 7+ services.

**Key principle:** Build resilience at the **API Gateway layer** first (rate limiting, global timeout), not per-service. Single place to handle failures.

## 6. Migration Roadmap

```
Phase 1: Database Migration (Month 1)
├── Set up PostgreSQL container (keep SQLite for read)
├── Update Drizzle config: sqlite → postgres dialect
├── Update better-auth adapter
├── Run expand-contract migration (dual-write → switch)
└── Verify all 14 tables + relations

Phase 2: Service Boundary Split (Month 2–3)
├── Extract Auth service (user, sessions, better-auth routes)
├── Extract Core service (transactions, categories, wallets, budgets, assets)
├── Extract Analytics service (dashboard aggregation, reports)
├── Nginx API Gateway routing
└── Update apiClient to point to gateway

Phase 3: Operational Maturity (Month 4+)
├── Add read replica for Analytics (Hybrid DB)
├── Redis for session/cache (if needed)
├── Structured logging ( pino / Winston)
├── Health check endpoints per service
└── Docker Compose profiles (dev vs prod)
```

**Timeline estimate:** Phase 1 is 1–2 weeks. Phase 2 is 4–8 weeks depending on full-time focus. Phase 3 is ongoing.

## Summary of Recommendations

| Area | Decision | Rationale |
|---|:---|:---|
| **DB strategy** | PostgreSQL, shared DB, multi-schema | ACID-safe, simple migration, low ops |
| **DB migration** | Expand-contract pattern | Zero downtime, Drizzle handles ORM |
| **Inter-service comms** | Sync REST via Nginx gateway | Simple, reliable, sufficient latency |
| **Service discovery** | Docker Compose DNS + Nginx | Zero extra infrastructure |
| **Resilience** | Timeouts + retry first; circuit breaker later | Build only what you need |
| **Out of scope** | gRPC, Kafka/RabbitMQ, per-service DBs, service mesh | Overkill for 1–3 person team, personal app |

The architecture above is designed to **evolve incrementally**. Start with Phase 1 (PostgreSQL swap), validate, then split services one at a time. The database design (shared PostgreSQL, schema-per-domain) allows service extraction without data migration — just move the service code and update the Nginx route.
