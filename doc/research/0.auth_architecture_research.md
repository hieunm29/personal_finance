# Multi-Service Authentication Architecture вҖ” Research Document

**Date:** 2026-03-22
**Author:** Claude (Security Architecture Research)
**Scope:** Preparing Personal Finance app for microservices decomposition

---

## Executive Summary

The Personal Finance app currently runs as a monolith with `better-auth` managing cookie-based sessions against a shared SQLite database. Splitting into microservices breaks this tight coupling in several ways. This document analyzes the impact and recommends a **hybrid JWT + API Gateway approach** вҖ” using JWTs for user-to-service authentication while keeping an API Gateway as the single enforcement point, with a shared Redis session store for revocation capability.

**Recommended architecture:** Option D (Session Forwarding) hybridized with Option B (Gateway), keeping `better-auth` at the gateway and issuing short-lived JWTs to downstream services.

## 1. Current Auth State: better-auth in Monolith

### How it works today

```
Browser
  в”Ӯ POST /api/auth/sign-in (credentials: email + password)
  в”Ӯ Cookie: better-auth.session_token=xxx; HttpOnly; SameSite=Lax
  в–ј
Hono Server (apps/server/src/index.ts)
  в”Ӯ
  в”ңв”Җв–ә /api/auth/**  вҶ’  better-auth handler (auth.ts:6-27)
  в”Ӯ                   в””в”Җв–ә DrizzleAdapter вҶ’ SQLite session table
  в”Ӯ                   в””в”Җв–ә Set-Cookie: session token
  в”Ӯ
  в””в”Җв–ә /api/*         вҶ’  auth middleware (index.ts:31-36)
                       в””в”Җв–ә auth.api.getSession({ headers })
                       в””в”Җв–ә c.set('userId', session.user.id)
                       в””в”Җв–ә next()
```

Key properties:
- **Session cookie** is `HttpOnly`, stored server-side in SQLite (`session` table вҖ” schema.ts:24-33)
- `auth.api.getSession()` reads the cookie token, hits SQLite on every call (with 5-minute cookie cache)
- Session is tied to the same SQLite DB where all app data lives
- No JWT involved вҖ” everything is server-side session verification

### What breaks in microservices

When you split into Auth Service + Core API Service + Asset Service + Reporting Service, each service currently needs its own copy of `better-auth` and its own session verification path:

```
в”Ңв”Җ Auth Service в”Җв”җ         в”Ңв”Җ Core API Service в”Җв”җ
в”Ӯ better-auth     в”Ӯ         в”Ӯ better-auth ??     в”Ӯ
в”Ӯ session in      в”Ӯ  Xв”Җв”ҖX   в”Ӯ same DB? different в”Ӯ
в”Ӯ SQLite A        в”Ӯ         в”Ӯ DB?                в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ         в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

Problems:

| Problem | Description |
|:---|:---|
| **Session fragmentation** | Each service needs its own SQLite with `user`/`session` tables, or a shared DB (coupling) |
| **Session verification latency** | Every service call requires a DB read to verify session |
| **Session revocation** | If user changes password or logs out, must invalidate across all service DBs |
| **CORS complexity** | Multiple services = multiple `trustedOrigins` configs |
| **`databaseHooks` coupling** | `seedUserData` trigger on user.create is in the auth module вҖ” needs to exist per-service or be extracted |

## 2. Auth Architecture Options Analysis

### Option A: Pure JWT (Stateless)

```
Login в”Җв”Җв–ә Auth Service в”Җв”Җв–ә JWT (15min) + Refresh Token (7d)
                    в”Ӯ
                    в–ј
User в”Җв”Җв–ә Core API в”Җв”Җв–ә Verify JWT signature (HS256/RS256)
      в”Җв”Җв–ә Asset  в”Җв”Җв–ә Verify JWT signature
      в”Җв”Җв–ә Report в”Җв”Җв–ә Verify JWT signature
```

**Pros:** No session DB, scales horizontally, each service validates independently.
**Cons:** Token revocation requires a blocklist (Redis), or you wait for TTL expiry. If a JWT leaks, it's valid until expiry. Refresh token rotation adds complexity.

**Verdict:** Too risky for personal finance. A leaked JWT gives full access until expiry. Revocation lag is unacceptable for financial data.

### Option B: API Gateway + Stateful Sessions

```
Browser в”Җв”Җв–ә API Gateway в”Җв”Җв–ә Auth Service (verify session in Redis)
                    в”Ӯ
              в”Ңв”Җв”Җв”Җв”Җв”Җв”јв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
              в–ј     в–ј      в–ј          в–ј
           Core   Asset  Report    Auth
           (trusts gateway header: X-User-Id)
```

**Pros:** Services are simple вҖ” no auth logic, just read `X-User-Id` header. Easy session revocation via Redis.
**Cons:** All trust is in the gateway. If gateway is compromised, all services are compromised. Internal network trust required.

**Verdict:** Good for internal networks, but single point of failure.

### Option C: OAuth 2.0 / OIDC

```
Browser в”Җв”Җв–ә Auth Service (AS) в”Җв”Җв–ә Authorization Code + PKCE
                    в”Ӯ
              в”Ңв”Җв”Җв”Җв”Җв”Җв”јв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
              в–ј     в–ј      в–ј          в–ј
           Core   Asset  Report    Auth
           (verify JWT via JWKS endpoint)
```

**Pros:** Industry standard, supports external IdPs, token introspection, scopes.
**Cons:** Significant complexity: authorization server, JWKS endpoint, token caching, scope enforcement. Overkill for a single-user personal finance app.

**Verdict:** The right choice for enterprise/multi-tenant. Too heavy here.

### Option D: Session Forwarding

```
Browser в”Җв”Җв–ә Auth Service в”Җв”Җв–ә Session stored in Redis
                    в”Ӯ
              в”Ңв”Җв”Җв”Җв”Җв”Җв”јв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
              в–ј     в”Ӯ      в”Ӯ          в”Ӯ
           Core в—„в”Җв”Җв”ҳ      в”Ӯ          в”Ӯ
           Asset в—„в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ          в”Ӯ
           Report в—„в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
           (all call Auth Service to validate session)
```

**Pros:** Central session store, instant revocation, compatible with cookie-based flow.
**Cons:** Every inter-service call hits Auth Service (latency), or services cache locally (complexity). Services still need to trust something.

**Why this works for Personal Finance:**

1. `better-auth` stays at the edge (Gateway) вҖ” no frontend changes, same cookie flow
2. Short-lived JWTs (10 min) limit exposure window if a token leaks
3. Each service validates JWT locally (no per-request Auth Service call)
4. Session revocation works: expire the session in Redis вҶ’ next JWT request fails
5. Services only need the signing secret (or public key for RS256), not the session DB

**Compromise vs. Option B:** Services don't blindly trust `X-User-Id` вҖ” they validate the JWT signature. This is the best balance of security and simplicity.

## 3. Auth Service Design

### Endpoints

```
POST   /auth/register     вҖ” create account вҶ’ seed user data
POST   /auth/login         вҖ” verify credentials вҶ’ session cookie
POST   /auth/logout        вҖ” delete session
GET    /auth/session       вҖ” get current session (for frontend)
POST   /auth/change-password
GET    /profile            вҖ” user profile
PUT    /profile            вҖ” update profile
```

### Password Security

Current state: `better-auth` handles password hashing (bcrypt via `bcryptjs`).

Hardening requirements for personal finance:

```typescript
// Recommended: add on top of better-auth
interface PasswordPolicy {
  minLength: 8,
  requireUppercase: true,      // or passphrase-based
  requireNumber: true,
  requireSpecial: false,        // special chars often reduce usability
  breachCheck: true,           // check against HaveIBeenPwned API (k-anonymity)
  maxAttempts: 5,              // lockout after 5 failed attempts
  lockoutDuration: 15 * 60,    // 15 minutes
}

// breachCheck: POST to HIBP API with k-anonymity
// GET /api/pwnedpasswords/range/{first5} вҶ’ compare k-anonymity suffix
```

### Shared Session Store (Redis)

```typescript
// Session structure in Redis
interface SessionStore {
  sessionId: string,          // better-auth token
  userId: string,
  authUserId: string,          // better-auth user.id
  profileId: string,          // userProfiles.id (used in all app tables)
  expiresAt: number,          // Unix timestamp
  userAgent: string,
  ipAddress: string,
}
```

### better-auth in Microservices

`better-auth` is primarily designed for monolith use. For microservices:

- **Gateway only:** Run `better-auth` only at the API Gateway. It issues the session cookie and validates it there.
- **Adapter pattern:** The Auth Service owns the `better-auth` configuration. Other services do NOT run `better-auth` вҖ” they trust the JWT from the gateway.
- **Database sharing:** The Auth Service owns the `user`, `session`, `account`, `verification` tables. Other services need only `userProfiles.id` (the app-level user ID) for data ownership checks.

**Migration path from current state:**

```
Phase 1: Introduce Redis
  - Add Redis as session cache (current SQLite still primary)
  - Session written to Redis on login
  - Auth middleware checks Redis first, falls back to SQLite
  - Zero frontend changes

Phase 2: Introduce Gateway
  - Deploy a thin Hono gateway in front of existing monolith
  - Gateway runs better-auth, issues JWT, forwards X-Internal-Token
  - Existing server validates X-Internal-Token instead of session cookie
  - Auth logic is now "gateway-owned"

Phase 3: Extract services
  - Split monolith into services
  - Each service receives JWT, validates with gateway's public key
  - better-auth is now only at gateway
```

## 4. Service-to-Service Authentication

For a personal finance app with a small number of services (5 max), a **static API key per service pair** is the minimum viable approach.

```
Core API Service в”Җв”Җв–ә Asset Service
  Header: X-Service-Key: <static-key-hash>
  Header: X-Service-Name: core-api
```

| Method | Complexity | Security | Recommendation |
|:---|:---|:---|:---|
| Static API keys | Low | Low (rotate manually) | **Minimum for this app** |
| mTLS certificates | High | Very high | Future upgrade |
| JWT service accounts | Medium | High | Recommended after initial split |

**Recommended: Static API keys with hashed storage**

```typescript
// Stored in environment variables, hashed in config
const SERVICE_KEYS = {
  'core-api':         hash(process.env.KEY_CORE_API),
  'reporting-service': hash(process.env.KEY_REPORTING),
  'auth-service':     hash(process.env.KEY_AUTH),  // gateway only
}

function validateServiceKey(name: string, key: string): boolean {
  return SERVICE_KEYS[name] === hash(key)
}
```

**Internal network assumption:** In Docker Compose, services communicate over a private `app-network`. External access is only via the gateway. This reduces the threat model significantly.

| Consideration | Direct Calls | Gateway |

| Security surface | Wider attack surface | Single enforcement point |

**For a personal finance app:** Security > minor latency cost. Use Gateway.

## 6. Frontend Authentication Flow

### Recommended: Cookie-based with JWT bridge

```
1. Login
   POST /api/auth/sign-in
   вҶҗ Set-Cookie: better-auth.session_token (HttpOnly, SameSite=Lax)

2. Every request (no code change)
   Browser automatically sends cookie

3. Gateway intercepts
   - Reads session cookie
   - Issues short-lived JWT (internal)
   - Forwards JWT to downstream services

4. Frontend redirect on 401
   (existing apiClient.ts has: 401 вҶ’ window.location.href = '/login')
```

This keeps the frontend **identical** to today вҖ” the cookie-based flow is preserved, and the JWT translation happens transparently at the gateway.

### JWT storage: NEVER in localStorage

```typescript
// WRONG вҖ” vulnerable to XSS
localStorage.setItem('token', jwt)

// CORRECT вҖ” JWT lives only in memory (gateway), not accessible to JS
// The browser only ever sees the session cookie (HttpOnly)
// Services receive JWT only via internal headers
```

### Refresh token rotation (optional future hardening)

```
Access token: 10 min TTL (JWT)
Refresh token: 7 day TTL (stored in Redis, tied to session cookie)

Flow:
  1. Access token expires вҶ’ API returns 401
  2. Frontend calls POST /auth/refresh (with session cookie)
  3. Gateway issues new access token
  4. Retry original request

Rationale: If session cookie is stolen, attacker gets 7 days.
  If they use it, it gets rotated and old one invalidated.
  Combined with IP/device tracking in session, this is strong.
```

## 7. Security Hardening Checklist

For personal financial data, these are non-negotiable:

### Rate Limiting

```typescript
// Gateway middleware
const rateLimits = {
  '/api/auth/sign-in':    { window: 15*60, max: 5,  by: 'ip' },
  '/api/auth/register':   { window: 60*60, max: 3,  by: 'ip' },
  '/api/auth/change-password': { window: 60*60, max: 3, by: 'userId' },
  '/api/*':               { window: 60,    max: 100, by: 'userId' },
}

// Use Redis for distributed rate limiting
// For single-node Docker Compose: in-memory rate limiter is fine initially
```

### Input Validation (already using Zod)

Continue using shared Zod schemas from `@pf/shared`. Ensure:
- All monetary amounts are positive integers (no negative, no decimals)
- Date formats are validated (YYYY-MM-DD regex)
- UUIDs match format before DB queries
- String lengths are bounded (note: max 500 chars, name: max 100)

```typescript
})
```

Store audit logs in a separate table or file (append-only, separate from app data).

### Data Encryption

| Layer | Current | Recommended |
|:---|:---|:---|
| **In transit** | HTTPS (Docker) | TLS 1.3, HSTS header |
| **At rest** | SQLite file (plaintext) | SQLCipher or filesystem encryption |
| **Backups** | None | Encrypted backups, stored off-site |
| **Session tokens** | SQLite (plaintext) | Redis with TLS + AUTH password |

**Minimum for launch:** Filesystem-level encryption (dm-crypt on host, or volume encryption in Docker). SQLite is a single file вҖ” if the server is compromised, the DB is readable. For personal finance, consider SQLite encryption via [SQLCipher](https://github.com/sqlcipher/sqlcipher) or migrate to an encrypted SQLite layer.
